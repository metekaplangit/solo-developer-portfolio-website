# Task Card — STEP-0023

- **ID:** STEP-0023
- **Type:** feature (visual — real product screenshots)
- **One outcome:** Show the five real Sole Focus marketing screenshots across the site — a full-width showcase gallery on the product detail page and the lead shot featured in the homepage spotlight — replacing the grey "Preview" placeholders, using build-time image optimization.
- **Why now:** User supplied the assets (`Final 5 Favorites`). This is research item C — the biggest visual lever — which was blocked on exactly these assets.
- **Acceptance criteria:** (1) The five 2880×1800 PNGs are optimized at **build time** via `astro:assets` (`<Image>` → responsive WebP with intrinsic width/height); originals live in `src/assets/sole-focus/screenshots/`, **not** `public/` (no full-size PNG shipped). (2) The detail page renders a **full-width showcase** — each screenshot in a rounded, soft-shadowed frame at 16:10, generous vertical rhythm, scroll-reveal (reduced-motion safe), **breaking out wider than the 40rem prose column**; the old placeholder `.shots` markup is gone. (3) Each image carries descriptive **alt text** (the baked-in headline is invisible to crawlers/AT, so alt describes the UI shown). (4) The homepage spotlight features the **lead** shot ("One quiet app for all of it") as a framed hero within the panel; layout stays responsive and the icon/name/CTA remain. (5) Below-the-fold images are `loading="lazy"`, the first is eager; **zero client JS** still shipped. (6) Content stays source-of-truth: the product `screenshots` array (alt text + order) drives rendering; images resolved via `import.meta.glob`. (7) `npm run build`, `npm run check`, `npm test` (+ a content test asserting 5 screenshots all have alt text), validator all pass; no horizontal overflow at mobile/desktop.
- **Non-goals:** Re-cropping or editing the screenshots (shown as-designed, full scene); a lightbox/zoom interaction; carousels or client-side sliders (no JS); screenshots for other products; adding `screenshot`/rating fields to the JSON-LD (possible later follow-up — kept out to stay lean).
- **Evidence method:** command/log (build/check/test/validator) + built-HTML/asset assertion that `dist/_astro/` contains generated WebP for all five shots and the detail + home HTML reference them with width/height + alt; runtime preview screenshots (desktop + mobile) of the detail showcase and homepage spotlight; no-overflow check.
- **Proof classification:** machine-verifiable (asset emission + HTML `<img>` attrs) + manual-runtime (visual preview).
- **Required checks:** `npm run build`; `npm run check`; `npm test`; validator; runtime preview (desktop + mobile).
- **Issue links:** closes research item C (hero/detail product screenshots).
